## 一、设计方案文档审计

### 1.1 总体评价

设计方案是一份超出一般"草案"成熟度的产品边界锁定文档。它不是模糊的愿景文档，而是已进入具体产品合同的阶段。

### 1.2 核心优势

1. **边界纪律极好** — 三大域、V1 必须做/降级做/不做三层划分清晰，每处都阻止了范围蔓延。
2. **对象模型完整** — 18 个核心对象覆盖了从导入到导出的全生命周期，命名和职责划分合理。
3. **证据体系是差异化亮点** — 证据密度规则、多状态管理、失效传播机制是 WriteStorm 的核心竞争壁垒。
4. **模块 × scope 模型正确** — 避免了"每个模块一个文件"的浅薄设计。
5. **风险自识别** — 第 27 节的五个风险、第 28 节的七个前置问题说明设计者了解关键不确定性。
6. **AI 可读摘要** — 第 29 节的 YAML 摘要是优秀实践，让技术方案可以直接引用。

### 1.3 关键缺陷

| 编号 | 缺陷 | 位置 | 严重度 |
|------|------|------|--------|
| D1 | **审查完成条件不可达** — 要求"所有用户可见 AI 资产必须确认、拒绝、排除或合并"，未定义审查分层和批量操作；长篇小说的对象数可能上千 | §25.1 | **High** |
| D2 | **账号式连接器无 fallback** — 要求"不能用 API Key 偷换验收"，但如果所有供应商都不支持合法本地 OAuth，V1 直接卡死，没有产品降级路径 | §6.4 | **Blocker** |
| D3 | **`paused_failed` 语义污染** — 一个状态混合了"暂停"和"失败"两种语义，且 AI 资产状态缺少"排除"状态但验收要求有"排除" | §8.1 / §8.2 | **High** |
| D4 | **`AnalysisModuleInstance` 未落为一等对象** — "模块 × scope"概念在文字中存在，但对象模型表、存储结构和验收条件中都没有它 | §7 / §18 / §25 | **High** |
| D5 | **证据失效策略偏技术字段、缺产品策略** — 定义了 offset/hash/structure 字段和失效规则，但未定义源文本是否默认不可变、用户能否替换源文件、失效后是否需要用户确认 | §12 | **High** |
| D6 | **专题视角被验收条件推成隐式新模块** — 文本强调了"不是新事实源"五次，但 V1 验收中把五个视角列为独立能力，实现上极易复制为独立事实 | §11.10 / §25.1 | **Medium** |
| D7 | **类型模板和标签定义过薄** — 6 个内置类型是好的起点，但每个类型的关注点、冲突优先级、默认模块配置都是空的；标签库明确"本轮不枚举" | §11.12 | **Medium** |
| D8 | **缺空/错/恢复态设计** — 主流程覆盖了成功路径，但首次启动、无 AI 连接器、导入损坏、任务中断恢复等路径缺少设计 | §9 / §21 | **Medium** |
| D9 | **桥段引用安全对象 `ProblemSolutionPattern` 字段过少** — 7 个字段，缺少"读者预期效果"和"与其他模式的组合关系" | §15 | **Low-Medium** |
| D10 | **缺少模块之间的版本依赖关系** — 人物模块引用情节事件，叙事模块引用人物声音，但模块重跑时下游依赖如何处理未定义 | §11.4–11.9 | **Medium** |
| D11 | **技法观察→候选→条目的去重策略缺失** — 同一技法在多本书中出现时，多个候选可能高度相似，V1 没有合并/消歧策略 | §14 | **Low-Medium** |
| D12 | **未定义"标记完成"后的生命周期** — 完成后的书籍是否可以重新打开、重新导入、追加源文本？当前状态机终结于 `completed`，缺少"已完成但需要补充分析"的状态 | §8.1 | **Low** |

## 二、审查文档审计

### 2.1 总体评价

审查文档系统性地覆盖了设计方案的关键问题，提供了具体修订方向和验收门禁。它不是泛泛的"这里有问题"，而是深入了产品逻辑层面的审查。

### 2.2 核心优势

1. **分层清晰** — 从排除项→证据限制→兼容性→产品设计问题→外部参考→同类软件反证→修订门禁，逻辑递进。
2. **严重度标注合理** — Blocker / High / Medium 分类与独立判断高度一致。
3. **精确的位置引用** — 每条问题标记了设计文档的具体行号，降低沟通成本。
4. **修订方向可执行** — 不是"需要改进"，而是"应该增加某概念、在某位置写入某规则"。
5. **7.5 节写回建议精准** — 每个修订点建议了插入设计文档的具体位置。
6. **门禁机制** — 7.4 节的 9 个技术方案前硬闸是高质量输出。

### 2.3 关键缺陷

| 编号 | 缺陷 | 位置 | 严重度 |
|------|------|------|--------|
| R1 | **未审设计文档的架构质量** — 审查聚焦于具体问题，没有评估设计文档整体的组织结构、信息冗余度（证据规则在 §11.11 和 §12 重复出现）、表述一致性 | 全文 | **Medium** |
| R2 | **未审三段式 AI 管线设计** — 第 10 章的管线是核心技术赌注，但审查没有质疑管线分段是否合理、story_segment 识别是否可行、全书归纳是否会丢失细节 | §10 | **Medium-High** |
| R3 | **未审对象模型的质量** — 18 个对象未有审查意见：`Perspective` 为何不在对象模型表中但信息架构中有？`ProblemSolutionPattern` 是拆解/原创桥接对象但存储位置未定义？`AIConstraint` 和 `SecretRef` 过于单薄？ | §7 | **Medium** |
| R4 | **未审模块间依赖与重跑级联** — 审查问了"模块 × scope 未落为一等对象"，但没追问模块重跑时如何影响下游模块的证据状态 | §11.4–11.9 | **Medium** |
| R5 | **缺少对"本地优先 + Windows 原生"假设的挑战** — 整个设计建立在 Windows 原生应用假设上，但竞品分析中的 Obsidian、Scrivener 都是跨平台。这个假设没被质疑 | §2 / §4 | **Low-Medium** |
| R6 | **竞品分析虽然精炼但深度不足** — 14 个外部软件只有 5 类共 4 节的简短分析，每类 3-4 句。对 Scrivener 的项目模板系统、Obsidian 的插件生态、NovelWriter 的伏笔池等关键差异点着墨不足 | §5 | **Low** |
| R7 | **未评估"模块正文可编辑"带来的审计风险** — 设计允许用户编辑模块的 Markdown 正文，但审查后用户修改正文可能绕过审查记录。审查没有注意到这个审计轨迹缺口 | §11 多处 | **Medium** |
| R8 | **未评估存储结构的合理性** — 建议的目录结构是否适合大规模使用？JSON 与 Markdown 双向同步的具体策略？审查没有触及 | §18 | **Medium** |
| R9 | **缺少修订优先级排序** — 7.2 节的 9 条修订清单没有划分优先级或依赖关系，技术方案团队不知道先改哪个 | §7.2 | **Low-Medium** |

## 四、共同盲点

两个文档都未触及的关键问题：

### 4.1 设计层遗漏

1. **模块重跑的下游级联** — 人物模块重跑后，引用人物的叙事模块、技法模块是否标记为 `needs_review`？证据是否失效？两个文档都没有定义。
2. **`TechniqueEntry` 的去重与消歧** — 多本书产出相似技法候选时，如何合并？V1"不做自动融合"但用户手动合并呢？
3. **标记完成后的再打开** — 书籍 `completed` 后，用户发现新证据或想追加分析，能否"反完成"？状态机终结于此。
4. **模块正文编辑的审计轨迹** — 用户编辑 Markdown 正文后，已审查的资产在文本上被修改了，但没有版本记录或重新审查的要求。

### 4.2 审查过程遗漏

5. **竞品分析深度不足** — 14 个外部工具只用了 4 节稀疏对照，没有拆解 Scrivener 的项目模板、Obsidian 的图谱、NovelWriter 的伏笔管理等与 WriteStorm 直接相关的机制。
6. **未引用非小说写作领域** — WriteStorm 定位"主要面向小说"，但非虚构/说明类写作的拆解需求可能完全不同。这个边界只在一句话中提及。

## 五、审计结论

### 5.1 总体评分

| 维度 | 设计文档 | 审查文档 |
|------|:--------:|:--------:|
| 覆盖度 | ★★★★☆ | ★★★★☆ |
| 深度 | ★★★★☆ | ★★★★☆ |
| 可执行性 | ★★★★☆ | ★★★★☆ |
| 边界纪律 | ★★★★★ | ★★★★★ |
| 自反思 | ★★★☆☆ | ★★★★☆ |
| **综合** | **4.0/5** | **4.2/5** |

### 5.2 核心判断

1. **设计文档是高质量草案** — 远超一般"草案"水平，已经接近可引用产品合同。它最大的优点是**边界纪律**，最大的已知风险是**审查负担不可达**和**连接器 fallback 缺失**。

2. **审查文档是高质量的审阅输出** — 精准、可执行、零误报。但它没有跳出产品层审视设计的可构建性和架构质量。

3. **两个文档互为良好补充** — 审查准确地找到了设计的关键薄弱点（Blocker × 1，High × 5，Medium × 3），并给出了可操作的修订清单。审查的 7.4 节（9 个硬闸）和 7.3 节（9 个补充验收用例）是高质量输出。
