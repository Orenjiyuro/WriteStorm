## 一、总体评价

### 1.1 第一印象

这是一份**异常详细**的工作拆分规划。18 个大块、150+ 个任务，每个任务都包含目标、范围、前置条件、验收标准和验证方法。其详细程度已经超出"工作拆分"的常规范畴——更接近一份**可执行的技术规格和工程路线图**。

**但有一个根本矛盾：** 这份规划的技术基础（Electron + Cross-platform + SQLite + Codex SDK）与设计方案草案的产品基础（Windows Native + JSON 主事实源）之间存在重大偏移，而规划本身没有说明这个偏移的来源和决策过程。

### 1.2 强度与主要风险

| 维度 | 评价 |
|------|------|
| 粒度 | ★★★★★ — 极细，每个任务可独立验证 |
| 依赖/顺序 | ★★★★☆ — 前置条件明确，但某些顺序值得商榷 |

## 二、与设计方案的偏移分析

这是本次审计发现的最重要问题。工作拆分规划在多个关键决策上与设计方案草案**不一致**，且没有说明这些偏移的出处。

| 维度 | 设计方案 (§§) | 工作拆分规划 (§0) | 偏移 | 影响 |
|:----:|:---|:---|:---|:---|
| **目标平台** | "Windows 原生" (§2, §4.1) | "Windows 11 + macOS 桌面应用；不做 Web 运行版" (§0) | **重大**：从 Win32 native → Electron cross-platform | 影响安全模型、文件系统访问方式、原生对话框、进程模型、打包方式 |
| **主事实源** | "JSON/结构化数据是主事实源" (§4.6, §18) | "SQLite 是唯一事务性主事实源" (§0) | **重大**：从 JSON-first → SQLite-first | 影响数据模型、迁移策略、导出格式、Markdown 编辑边界 |
| **AI 连接器** | "需实现一个账号式连接器；API Key 可以作为支持能力" (§6.4) | "V1 AI 只允许 Codex SDK" (§0) | **不一致**：设计接受 API Key 作为次要路径，规划完全锁定 Codex | 风险更高：Codex 可能不通过，但设计文档的 API Key fallback 被排除 |
| **Markdown 边界** | "Markdown 是阅读和正文编辑视图" (§4.6) | "Markdown 只承载 AnalysisModuleInstance.body" (§0) | **收敛**：规划进一步收紧了 Markdown 的范围 | 更严格，更安全 |
| **导出格式** | "JSON 是导出格式的一部分" (§19.1) | "JSON/SQLite-free 机器包" (§18.19) | **轻微偏移**：规划说"不把 JSON 当主事实源反写" | 与设计文档兼容 |

### 2.1 未解决的张力

1. **Windows Native → Electron Cross-platform 的决策没有追溯**。设计方案的所有设计假设（本地优先、Windows 原生壳、Windows 安全存储、native dialog）都基于 Windows 原生路径，但规划切换到了 Electron 却没有在新的上下文中重新验证这些假设。

2. **SQLite → JSON 的角色转换没有讨论**。设计方案明确说 JSON 是主事实源，Markdown 是编辑视图。规划改为 SQLite 是唯一事务性事实源，JSON/Markdown 只做导出。这个改变对证据引用、版本管理、导出包、资料库修复都有深远影响，但规划没有解释为什么需要这个改变。

3. **Codex SDK 锁定 + No Fallback 的决策比设计文档更激进**。设计方案在 §6.4 说"API Key 可以作为支持能力"——虽然不能替代账号式连接器验收，但可以作为降级路径。规划在 §0 直接说 "SDK spike 不通过则 AI 阻塞，不回退 `codex exec`、app-server、API key 直连、本地模型或其他供应商"——这比设计文档更严格。

### 3.2 依赖链风险评估

#### 风险 1：Codex SDK Spike（大块 13）位置过晚

大块 13 排在第 13/18 位，前置包括大块 1、2、10、12。这意味着：**在完成大块 1-12 和 14-16 之后，才发现 Codex SDK 不可用**。

- 如果 spike No-Go，大块 1-12（Foundation）和大块 14-16（Content Model Depth）的产出将无法对接 AI 管线
- 大块 17 要求 14-16 完成，而 14-16 需要 13 的前置，但 14-16 的完成不等待 13 Go——这意味着 13 未 Go 时 14-16 的工作会白做
- **建议**：把大块 13 提前到至少大块 6 之后、8 之前。这样在进入结构识别和模块工作台之前就知道了 AI 可行性

#### 风险 2：Foundation 阶段 12 个大块的累积风险

大块 1-12 都是 Foundation，不产出任何用户可感知的 AI 能力。一个用户在 V1 的真实使用路径（导入→拆解→审查→完成）到大块 18 才完整闭合。这意味着：

- 在完成大块 12 之前，用户无法做任何实际的小说拆解
- 大块 6（SQLite）之前的 5 个大块都是纯契约/类型代码，没有真实数据持久化
- **这可能导致"大量契约代码，最后发现数据模型不对"的返工风险**

#### 风险 3：大块 14-16 与大块 13 的时序依赖

大块 14 的前置写的是"大块 3、6、9"——并不包含大块 13。这意味着：
- 大块 14（七模块深度规格）可以在 Codex SDK spike 之前开始
- 如果 spike No-Go，大块 14 产出的 schema 和规格无法用 AI 填充
- **建议**：大块 14 加前置条件"大块 13 Go 或已确认 AI 策略"

| 编号 | 盲点 | 位置 | 严重度 |

## 六、交叉审计总结

### 6.1 三份文档的关系

```
设计方案草案 (2026-07-04)
    ↓ 审查发现问题
审查问题记录 (2026-07-05)
    ↓ 回应审查 + 技术决策
工作拆分规划 (未注明日期)
```

工作拆分规划是**第一份进入技术选层和技术细节的文档**。它做了三个前两份文档没有做的重要决策：
1. Electron + Cross-platform 替代 Windows Native
2. SQLite 替代 JSON 作为主事实源
3. Codex SDK 锁定替代账号式连接器

**这些决策本身可能都是合理的，但问题在于：它们没有被追溯回设计方案，也没有在任何地方解释 tradeoff 和决策理由。**

## 七、建议

### 7.1 立即处理

1. **补充技术决策记录** — 在规划或单独文档中记录：为什么选 Electron 而非 Windows Native？为什么选 SQLite 而非 JSON？为什么锁 Codex SDK 而不留 fallback？每个决策的备选方案和 tradeoff 是什么？

2. **重新审视大块 13 的位置** — 建议将 Codex SDK Spike 从第 13 位提前到至少第 7 位之后（即大块 6 SQLite 完成后立即做），避免 foundation 阶段的大量工作在 spike No-Go 时被浪费。

3. **明确 Codex SDK No-Go 的应急路径** — 规划中"完全阻塞"是一个回答，但需要产品层面的决策：如果 Codex 不可用，V1 是延期、换方案、还是裁剪 AI 能力到一个更小的范围？
